# Pebble UI — Design Systems Starter

A course project scaffold that runs the full professional design-to-code pipeline:

**Figma variables → Tokens Studio → `tokens.json` → Style Dictionary → CSS custom properties → React components → Storybook**

---

## Quick start

```bash
npm install
npm run storybook     # open component library at localhost:6006
npm run dev           # open kitchen-sink demo at localhost:5173
```

## The token update loop

This is the core workflow the assignment is teaching:

1. **Change a value** in `src/tokens/tokens.json`
   (or update your Figma variables and re-export via Tokens Studio)

2. **Regenerate CSS** by running:
   ```bash
   npm run tokens
   ```

3. **Observe** — every component that uses that token updates automatically.
   No component code changes required.

Try it: change `--pebble-global-color-purple-600` to a different hex, run `npm run tokens`, and watch every primary Button, focused Input border, and brand Badge update in both the demo app and Storybook.

---

## Token architecture

Tokens are organised in three layers:

| Layer | Purpose | Example |
|---|---|---|
| **Global** | Raw values — the full palette | `color.purple.600 = #534AB7` |
| **Semantic** | Named roles that reference globals | `color.brand.default → {global.color.purple.600}` |
| **Component** | *(student extension)* Component-specific overrides | `button.padding.md = {global.spacing.4}` |

Components must **only** use semantic tokens (the `--color-*`, `--space-*`, `--radius-*` shorthand aliases in `tokens.css`). No hardcoded hex values in component CSS — ever.

## Storybook workflow

```bash
npm run storybook
```

Each component has stories for:
- **Default** — the standard happy path
- **Variants** — all visual flavours side by side
- **States** — disabled, loading, error, etc.
- **All sizes** — sm / md / lg together

### Addons to use

| Addon | What to check |
|---|---|
| **Controls** | Tweak props live — verify all combinations look correct |
| **a11y** | Flags WCAG contrast failures that trace back to token choices |
| **Backgrounds** | Test on white, subtle, and dark surfaces |
| **Docs / Autodocs** | Auto-generated API reference from JSDoc + stories |

## Assignment brief — for students

### Overview

You will replace the starter token values with your own design system and extend the component library with at least one new component. Every visual decision must be traceable back to a token.

---

### Part 1 — Figma (week 1)

**Deliverable:** Figma file with components and a token set

1. Set up a Figma file using Figma Variables (the native Variables panel, not just styles).
2. Define at minimum:
   - **Color:** a primary brand color with 3 stops (subtle / default / hover), a surface color, and two text colors (primary / muted)
   - **Spacing:** at least 4 steps on a consistent scale (e.g. 4 / 8 / 16 / 24 px)
   - **Radius:** at least 2 steps (sm / md)
   - **Typography:** 2 font sizes, 2 weights
3. Design all four components (Button, Card, Input, Badge) in Figma using your variables — no hardcoded values in the component file.
4. Document at least two modes: **light** and **dark** (or two brand themes).

**Evaluation criteria**
- Variables are named semantically (`color/brand/default`, not `purple`)
- Components in Figma match the token values exactly
- Light/dark modes switch correctly using variable modes

---

### Part 2 — Tokens (week 2)

**Deliverable:** Updated `tokens.json` + regenerated `tokens.css`

1. Install the **Tokens Studio for Figma** plugin.
2. Map your Figma Variables to a Tokens Studio token set.
3. Export to `tokens.json` and replace the starter file.
4. Run `npm run tokens` and verify `tokens.css` reflects your values.
5. Open Storybook — your components should now reflect your design.

**Evaluation criteria**
- `tokens.json` follows the global → semantic layering convention
- No hardcoded hex values remain in any component CSS
- Running `npm run tokens` produces a valid `tokens.css` without errors

---

### Part 3 — Storybook (week 3)

**Deliverable:** Stories for all components + one new component

1. Review the existing stories. Add any missing states you discover.
2. Build one additional component of your choice (suggestions: `Avatar`, `Alert`, `Toggle`, `Tooltip`, `Select`).
   - Component file in `src/components/YourComponent/`
   - CSS using only token variables
   - Stories covering all variants and states
   - a11y addon must show no violations
3. Check the **a11y** addon for all four original components with your token values. Fix any contrast failures by adjusting the token (not by overriding in component CSS).

**Evaluation criteria**
- New component follows the same file structure as existing ones
- All stories render without console errors
- a11y addon passes WCAG AA for all stories
- Controls panel works for all controllable props

---

### Part 4 — Final project (week 4)

**Deliverable:** A 2–3 screen React app built entirely from your component library

Requirements:
- Every UI element uses a component from `src/components/` — no one-off inline styles for color, spacing, or radius
- At least one screen includes a form (Input + Button)
- At least one screen displays a list of Cards
- The app must work with both your light and dark token modes (swap a CSS class or use `prefers-color-scheme`)

Suggested app: a simple product listing page with a search input, a grid of product Cards (with Badge for status), and a detail view triggered by clicking a Card.

**Evaluation criteria**
- No hardcoded color, spacing, or radius values outside `tokens.css`
- Visual consistency — spacing and color feel systematic, not ad hoc
- Changing one semantic token (e.g. `color.brand.default`) updates the whole app
- Code is readable and component props are used correctly

## Extending the token system

To add a dark mode:

```css
/* in tokens.css, add: */
@media (prefers-color-scheme: dark) {
  :root {
    --color-brand:          #7F77DD;
    --color-surface:        #1a1a18;
    --color-surface-subtle: #2C2C2A;
    --color-text:           #F1EFE8;
    --color-text-muted:     #B4B2A9;
    --color-border:         #444441;
    --color-border-strong:  #5F5E5A;
  }
}
```

Or wire a `.theme-dark` class to a toggle and set those overrides on that selector instead.
